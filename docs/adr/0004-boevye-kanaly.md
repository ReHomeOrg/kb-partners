# ADR-0004 — Боевые каналы доставки и уведомлений («разрабатываем сами»)

- Статус: Принято (решение Архитектора, 2026-06-11)
- Дата: 2026-06-11
- Контекст: ТЗ `kb-partners` v1.1 §4.9, §9.2, E4/E5/E8 (FR-4.2/5.1/8.1/8.2), §16.10; CLAUDE.md правило 6

## Контекст

ТЗ требует боевые каналы доставки заявки партнёру (§9.2: PartnerApi/CRM/Telegram/MAX/
Email) и боевые уведомления заявителю/партнёру/оператору (E8: push/SMS/email). Правило 6
запрещает новые внешние SDK без ADR; принцип «разрабатываем сами»; все ПДн — в РФ (ФЗ-152).

## Решение

**Все каналы — собственные адаптеры по опубликованным HTTP-контрактам провайдеров, без
вендорских SDK.** Каждый адаптер: config-gated (пустой креденшл → инертен), resilient
(timeout→breaker→retry, `ResilientHttpClient`), секреты — ссылкой на kb-vault, ПДн в логи
не пишет.

### Уведомления (E8)

| Канал | Провайдер | Реализация |
|---|---|---|
| SMS | **SMS.ru** (РФ) | `notifications/channels.py::send_sms` — POST `/sms/send` (api_id, json=1) |
| Email | **SMTP** (РФ) | `send_email` — stdlib `smtplib` STARTTLS в `asyncio.to_thread` |
| Push | **Web-push (VAPID)** | seam; полная реализация — M11 (store подписок + SW портала) |

Контакт адресата (ПДн) резолвится **на дрейне** (`notifications/contacts.py`): заявитель →
rehome.one (`get_requester_context`), партнёр → kb-platform (`get_partner_contact`),
оператор → `notify_operator_email`. В outbox контакт НЕ кладётся.

### Каналы доставки партнёру (E4/E5)

| Канал | Провайдер | Статус |
|---|---|---|
| PartnerApi | подписанный webhook | готов (M3) |
| Telegram | Telegram Bot API (HTTP) | M11.1 — свой адаптер, без SDK |
| MAX | MAX Bot API (HTTP) | M11.1 — `[ДОПУЩЕНИЕ §16.10]`, по аналогии с Telegram; сверить возможности |
| Email | SMTP исходящий + IMAP-парсер | M11.2 — IMAP через stdlib `imaplib` |
| CRM | **Bitrix24 + amoCRM** (REST) | M11.3 — свои адаптеры, OAuth/webhook |

Новый канал = реализация `DeliveryChannel` + запись `PartnerChannelConfig`, без правки ядра.

## Последствия

- Боевая доставка включается креденшлами в конфиге; по умолчанию каналы инертны.
- Замена/добавление провайдера = новый адаптер + env, без правки сервисного слоя.
- MAX Bot API — допущение (§16.10): адаптер строится по контракту Telegram-стиля; при
  расхождении правится только адаптер.
- Стоимость/латентность боевых каналов — на ops-мониторинге (`external_client_*`).
