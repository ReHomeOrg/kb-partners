# ТЗ: модуль обработки партнёрских заявок reHome — `kb-partners`
### Версия 1.1 · 2026-06-11 · для разработки Клодом (Claude Code), двухагентная схема

> **Назначение.** Техническое задание на новый модуль экосистемы reHome, который принимает заявки пользователей на партнёрские услуги, **распознаёт категорию заявки** (какой продукт и какому партнёру передать), **автоматически диспетчеризует** заявку выбранному партнёру по настроенному каналу взаимодействия и ведёт её жизненный цикл до приёмки и расчёта. Вместе с модулями **KB (kb-platform)** и **kb-support**, под управлением **ИИ-агента-оркестратора**, модуль обеспечивает максимально возможную автоматизацию работы с заявками в reHome.
>
> ТЗ опирается на «Бриф ТЗ: партнёрский модуль reHome (`kb-partners`)» и архитектурное описание reHome (KB + kb-support). Документ построен **contract-first** и снабжён проверяемыми acceptance-критериями — под двухагентную схему разработки (агент-исполнитель + агент-ревьюер), как при создании KB и kb-support.
>
> **Статус решений:** часть архитектурных развилок зафиксирована **рекомендуемыми дефолтами** (помечены `[ДЕФОЛТ — подтвердить]`). До старта реализации их подтверждает Архитектор (сводка — §16).
>
> **Изменения v1.1.** Зафиксировано имя оркестратора — «Консьерж» (`kb-concierge`, ТЗ-компаньон). Добавлено: переходы «партнёр отклонил» и «отмена заявки» в машине состояний; правовое основание передачи ПДн партнёру (152-ФЗ, поручение на обработку); делегирование прав пользователя при действиях агента (on-behalf-of); eval-гейт классификатора; идемпотентность создания `ServiceOrder`; конвенции API (Idempotency-Key, пагинация), rate-limiting, ретенция ПДн, ориентиры SLO.

## 1. Контекст и цель

### 1.1 Бизнес-суть
Пользователь-наниматель запрашивает услугу платного партнёра (финансовая **группа B**: клининг, переезд, мелкий ремонт, доставка ключей). Модуль оркестрирует жизненный цикл заявки-на-услугу:

```
приём → распознавание категории/партнёра → подбор/назначение → диспетчеризация партнёру
→ принятие партнёром → выполнение → приёмка → оплата/escrow/комиссия → отзыв
```

### 1.2 Что модуль добавляет к экосистеме
Доменная модель партнёров (`Collaborator`, `ServiceOrder`) уже существует в kb-platform. Новый модуль добавляет то, чего сейчас нет:
1. **Мультиканальный приём** заявок от пользователей.
2. **AI-распознавание категории** заявки (продукт + кандидат-партнёр) из свободного ввода.
3. **Автоматическую диспетчеризацию** — передачу заявки конкретному партнёру по **настроенному каналу** (API / CRM / Telegram / MAX / e-mail).
4. **Приём ответов партнёра** обратно по тем же каналам и продвижение заявки по статусам.
5. **Agent-ready** контур: модуль управляется ИИ-агентом-оркестратором с безопасным human-handoff в kb-support.

### 1.3 Рабочее имя
Сервис `kb-partners` (`[ДЕФОЛТ — подтвердить]`, §4.1).

## 2. Глоссарий

- **Партнёр** — внешний исполнитель услуги. В коде — сущность `Collaborator` (kb-platform), 14 типов, 4 финансовые группы. Целевые для модуля — группа B.
- **Заявка-на-услугу (`ServiceRequest`)** — центральная сущность модуля; «тикет с партнёрской ногой».
- **`ServiceOrder`** — заказ услуги у партнёра группы B в kb-platform (комиссия + escrow). Модуль порождает/обновляет его по API.
- **Канал доставки (DeliveryChannel)** — способ передачи заявки партнёру: API / CRM / Telegram / MAX / e-mail.
- **Классификатор (RequestClassifier)** — AI-компонент распознавания категории/параметров заявки.
- **Подбор (PartnerMatcher)** — выбор конкретного партнёра под категорию/гео/рейтинг/доступность.
- **Оркестратор «Консьерж» (`kb-concierge`)** — ИИ-агент-дирижёр экосистемы, управляющий заявками поверх KB + kb-support + kb-partners (отдельный сервис; ТЗ-компаньон).
- **`access_level`** — уровень доступа ресурса (PUBLIC/LOGGED/AGENT/STAFF/LEGAL/HR_RESTRICTED), фильтр на хранилище.
- **`scope`** — права субъекта, вычисляются бэкендом из проверенного JWT.
- **m2m** — server-to-server через OAuth2 Client Credentials (Keycloak).
- **outbox / idempotency** — паттерны гарантированной и неповторяющейся доставки.

> **Терминология.** В прозе и в пользовательских текстах — «партнёр» (с ё, все падежи). Английские идентификаторы кода/контрактов (`Collaborator`, `CollaboratorReview`, `collaborator_id`, `fallback_collaborator_id`, `ServiceOrder`) сохраняются как есть — это контракты API/кода.

## 3. Границы (scope)

**Входит в модуль:**
- Доменная сущность `ServiceRequest` и её машина состояний.
- Мультиканальный приём: ЛК-форма, AI-чат (`from-chat`), эскалация из тикета поддержки (`from-ticket`), входящий API.
- AI-распознавание категории и параметров заявки (классификатор) с порогом уверенности и human-handoff.
- Подбор/назначение партнёра (авто + ручное оператором) и fallback-цепочка.
- Диспетчеризация по настроенному каналу (API / CRM / Telegram / MAX / e-mail) — отправка и приём ответа.
- SLA на принятие заказа партнёром и на выполнение, с fallback.
- Приёмка работ, флоу спора (DISPUTE), связь с претензионным контуром kb-support.
- Связь с платёжным контуром rehome.one (escrow/комиссия — только ссылки/триггеры).
- Отзыв/рейтинг партнёра (переиспользование `CollaboratorReview`).
- Уведомления пользователю и партнёру; исходящие webhooks.
- Интеграция с ИИ-агентом-оркестратором (tools + события).
- Портал партнёра LIGHT (MVP).

**НЕ входит (делают смежные системы):**
- **Расчёт денег** — комиссия (10–25 %), escrow, выплата партнёру считает и проводит **платёжный контур rehome.one**. Модуль хранит заявку, решения и ссылки, деньги **не считает**.
- **Источник истины** по людям/договорам/объектам — rehome.one (читается по API).
- **Реестр партнёров как мастер-данные** — kb-platform (`[ДЕФОЛТ — подтвердить]` владение `ServiceOrder`, §4.1).
- **Претензионный разбор** компенсаций/гарантий — контур kb-support (модуль с ним стыкуется, не дублирует).
- **Сам ИИ-агент-оркестратор** («Консьерж» / `kb-concierge`) как рантайм — отдельный сервис уровня экосистемы (отдельное ТЗ-компаньон); модуль обязан быть **agent-ready**, но не реализует агента внутри себя (§4.10).

## 4. Принятые решения по архитектурным развилкам

Дефолты к подтверждению Архитектором (см. сводку §16).

**4.1 Отдельный сервис · владение `ServiceOrder`.** `[ДЕФОЛТ]` Отдельный сервис `kb-partners`: своя БД, репозиторий, деплой. Реестр партнёров остаётся в kb-platform; **владение жизненным циклом `ServiceOrder` переходит к `kb-partners`** (он — оркестратор заказа), kb-platform остаётся источником реестра. Контракт чтения реестра + создания/обновления заказа фиксируется в OpenAPI.

**4.2 Деньги и escrow.** `[ИНВАРИАНТ]` Модуль хранит заявку, решение, ссылки; расчёт комиссии/escrow/выплат — платёжный контур rehome.one. Контракт `BankProvider`/escrow — провизорный за adapter.

**4.3 SLA принятия/выполнения и fallback.** `[ДЕФОЛТ]` Таймер на принятие заявки партнёром; при breach — авто-`fallback` на следующего партнёра в цепочке; при исчерпании цепочки — статус `FAILED_DISPATCH` и эскалация оператору. Параметры (длительности, число попыток) — конфигурируемые на категорию/партнёра.

**4.4 Портал партнёра.** `[ДЕФОЛТ]` MVP — **LIGHT** (push/email + приём/отклонение/смена статуса через канал/API, минимальный экран списка заявок). FULL-дашборд — следующая итерация.

**4.5 Подбор партнёра.** `[ДЕФОЛТ]` Авто-подбор по: категория → гео (`service_area`) → доступность (расписание/нагрузка) → рейтинг → готовность канала. Ручное переназначение оператором/агентом доступно всегда.

**4.6 Приёмка и спор.** `[ДЕФОЛТ]` Приёмку подтверждает пользователь; при споре заявка уходит в `DISPUTE` и при необходимости порождает претензию `COMPENSATION` в kb-support.

**4.7 Каналы инициации (intake).** `[ДЕФОЛТ]` MVP: ЛК-форма + AI-чат (`from-chat`) + эскалация из тикета (`from-ticket`) + входящий API. Единый паттерн приёма.

**4.8 Отзывы/рейтинг.** `[ИНВАРИАНТ]` Переиспользовать `CollaboratorReview`/rating kb-platform, не заводить свою систему.

**4.9 Уведомления.** `[ДЕФОЛТ]` Переиспользовать паттерн диспетчера уведомлений kb-support (push/SMS/email), боевая доставка config-gated.

**4.10 Расположение ИИ-агента-оркестратора.** `[РЕШЕНО]` Агент-рантайм — отдельный сервис **«Консьерж»** (`kb-concierge`, отдельное ТЗ-компаньон); `kb-partners` предоставляет ему **инструменты** (идемпотентные API-эндпоинты + структурированные схемы) и **события** (webhooks), но агента внутри себя не держит. Альтернатива (агент внутри модуля) отклонена по арх-константе слабой связанности.

**4.11 Распознавание категории.** `[ДЕФОЛТ]` Классификатор на `LLMProvider` (YandexGPT/GigaChat/vLLM/Mock, как в kb-search) + детерминированные правила (ключевые слова/каналы) как быстрый путь. Порог уверенности; ниже порога или при неоднозначности → `NEEDS_REVIEW` (human-handoff).

## 6. Доменная модель

Все сущности несут `access_level`; фильтрация на уровне хранилища; недоступный ресурс → 404.

### 6.1 `ServiceRequest` — заявка-на-услугу
| Поле | Тип | Назначение |
|---|---|---|
| `id` / `number` | UUID / str | ID и человекочитаемый номер |
| `requester_id` | str | пользователь (rehome.one) |
| `booking_id` / `premises_id` | str? | контекст договора/объекта |
| `channel_in` | enum | WEB_FORM / AI_CHAT / SUPPORT_TICKET / API / MESSENGER_INBOUND |
| `source_ref` | JSONB | chat_session_id / ticket_id / message_id источника |
| `raw_input` | text | исходный запрос пользователя |
| `raw_input_masked` | text | ПДн-маскированная копия для LLM/логов |
| `category` | enum | CLEANING / MOVING / REPAIR / KEY_DELIVERY / OTHER |
| `product_code` | str? | уточнённый продукт внутри категории |
| `classification` | JSONB | {confidence, model, version, params{}, classified_at, method∈{rules,llm}} |
| `partner_id` | str? | выбранный `Collaborator` |
| `match_trace` | JSONB | кандидаты + причины ранжирования |
| `fallback_chain` | JSONB | упорядоченный список `collaborator_id` для авто-fallback |
| `delivery_channel` | enum? | разрешённый канал выбранного партнёра |
| `service_order_id` | str? | ссылка на `ServiceOrder` (kb-platform) |
| `status` | enum | см. §7 |
| `sla` | JSONB | accept_deadline / perform_deadline / paused_intervals / breach_flags / sla_state |
| `amount_ref` / `escrow_ref` | str? | ссылки контура (не суммы для расчёта) |
| `dispute_id` / `claim_ref` | str? | связь с kb-support |
| `rating_ref` | str? | `CollaboratorReview` |
| `custom_fields` | JSONB | прочее; служебные ключи редактируются для не-операторов |
| `access_level` | enum | двухконтурность |
| `idempotency_key` | str | дедуп приёма |
| `created/updated/dispatched/accepted/done/accepted_by_user/paid_at` | ts | таймстемпы |

### 6.2 `RequestMessage`
`author_type` (requester/operator/partner/system/ai), **`is_internal`** (внутренняя заметка невидима пользователю и партнёру — критичный инвариант), text, attachments[] (kb-files), `created_at`.

### 6.3 `RequestHistory`
Неизменяемый аудит: `actor_id` (включая сервис-принципал ИИ-агента), `action`, `from_value`/`to_value`, `ts`.

### 6.4 `PartnerChannelConfig` — конфигурация канала на партнёра
| Поле | Назначение |
|---|---|
| `collaborator_id` | партнёр |
| `channel_type` | API / CRM / TELEGRAM / MAX / EMAIL |
| `priority` | порядок предпочтения при нескольких каналах |
| `config` (JSONB) | endpoint/CRM-тип/chat_id/email/шаблоны; **секреты — ссылкой на kb-vault, не инлайн** |
| `inbound_token` | секрет для верификации входящих (HMAC) |
| `is_active` / `health` | активность и состояние healthcheck |

### 6.5 `DispatchAttempt`
`request_id`, `channel_type`, `attempt_no`, `status` (SENT/DELIVERED/FAILED/ACK), `provider_response` (без ПДн), `idempotency_key`, `outbox_ref`, `ts`.

### 6.6 `AutomationRule`
Правила `on_create` (классификация→подбор→диспетчеризация) и `time_based` (таймеры/fallback/эскалация). Условия по category/channel/гео/scope.

## 7. Машина состояний `ServiceRequest`

Бэкенд отдаёт `allowed_transitions`; фронт логику не дублирует. Каждый переход — запись в `RequestHistory`.

```
NEW → CLASSIFYING → CLASSIFIED → MATCHING → ASSIGNED → DISPATCHED → ACCEPTED
    → IN_PROGRESS → DONE → ACCEPTED_BY_USER → PAID

ветвления:
  CLASSIFIED → NEEDS_REVIEW          (низкая уверенность/неоднозначность; human-handoff)
  NEEDS_REVIEW → MATCHING            (оператор/агент подтвердил категорию/партнёра)
  DISPATCHED → MATCHING              (партнёр отклонил/не принял в срок → следующий в fallback-цепочке)
  DISPATCHED → FAILED_DISPATCH       (исчерпана fallback-цепочка → эскалация оператору)
  FAILED_DISPATCH → MATCHING         (ручное переназначение)
  DONE|ACCEPTED_BY_USER → DISPUTE    (спор → возможна претензия COMPENSATION в kb-support)
  <любой нетерминальный> → CANCELLED (с причиной)
```

Терминальные: `PAID`, `CANCELLED`, `REJECTED` (если претензия отклонена и заявка закрыта). Запрещённый переход → `409`.

## 8. Функциональные требования (эпики)

### E1. Приём заявок (мультиканальный intake)
- **FR-1.1** Приём из ЛК-формы (браузерный JWT, `aud: kb-partners`).
- **FR-1.2** `POST /requests/from-chat` (SERVICE-only, из kb-search): создаёт заявку с `channel_in=AI_CHAT`, переносит транскрипт в `source_ref`. Идемпотентность по `chat_session_id`.
- **FR-1.3** `POST /requests/from-ticket` (SERVICE-only, из kb-support): эскалация «вызвать клининг/переезд/ремонт» из тикета → заявка с обратной ссылкой.
- **FR-1.4** Входящий API (m2m) для партнёрских/внешних инициаторов.
- **FR-1.5** Вложения через kb-files (фото объёма работ).
- **FR-1.6** При приёме формируется `raw_input_masked` (ПДн замаскированы) до любых LLM-вызовов и логов.

### E2. Распознавание категории (AI-классификатор)
- **FR-2.1** Из `raw_input_masked` определить `category` (+ `product_code`) и извлечь структурированные `params` (адрес/площадь/дата/объём и т.п.).
- **FR-2.2** Быстрый детерминированный путь (правила: канал/ключевые слова) → при однозначности минуя LLM.
- **FR-2.3** LLM-путь через `LLMProvider` (переключение по env, как в kb-search), с версионированием модели/промпта в `classification`.
- **FR-2.4** Порог уверенности `confidence_threshold` (конфиг). Ниже порога / неоднозначно → `NEEDS_REVIEW`.
- **FR-2.5** Реклассификация по запросу оператора/агента (`POST /requests/{id}/classify`).
- **FR-2.6** Полная трассируемость: метод, модель, версия, уверенность — в `classification` и в истории.

### E3. Подбор партнёра (matching + fallback)
- **FR-3.1** Кандидаты из реестра kb-platform по `category` + `service_area` (гео) + доступность + рейтинг + готовность канала (`is_active`/health).
- **FR-3.2** Ранжирование и `match_trace` (объяснимость выбора).
- **FR-3.3** Формирование `fallback_chain`.
- **FR-3.4** Ручное назначение/переназначение оператором или агентом (`POST /requests/{id}/assign`).
- **FR-3.5** Создание/привязка `ServiceOrder` в kb-platform при назначении — **идемпотентно** (повтор/ретрай не создаёт дубль заказа; ключ идемпотентности на стороне вызова).

### E4. Диспетчеризация и каналы доставки
- **FR-4.1** По выбранному партнёру разрешить `delivery_channel` из `PartnerChannelConfig` (с учётом priority/health).
- **FR-4.2** Передать заявку партнёру через адаптер канала (см. §9): API / CRM / Telegram / MAX / e-mail.
- **FR-4.3** Каждая попытка — `DispatchAttempt`; доставка идемпотентна, через outbox; ретраи с backoff.
- **FR-4.4** SLA на принятие; при неответе/отказе — авто-fallback на следующего из `fallback_chain`.
- **FR-4.5** Исчерпание цепочки → `FAILED_DISPATCH` + эскалация оператору (и событие агенту).
- **FR-4.6** Шаблонизация исходящего сообщения партнёру по каналу; передаётся **минимально необходимый** объём ПДн для оказания услуги. Передача ПДн партнёру — на правовом основании 152-ФЗ (**поручение на обработку ПДн** в партнёрском договоре); состав передаваемых полей фиксируется по категории услуги.

### E5. Приём ответов партнёра (inbound)
- **FR-5.1** Приём статусов/ответов партнёра обратно по каналу: webhooks (API/CRM), bot-updates (Telegram/MAX), IMAP-парсер (e-mail).
- **FR-5.2** Входящие — подписанные, SERVICE/токен-верифицированные, идемпотентные, с защитой от replay (timestamp/nonce); маппинг на `ServiceRequest` по корреляционному ключу.
- **FR-5.3** Продвижение статуса: ACCEPTED / **отклонение** / IN_PROGRESS / DONE; ответы партнёра как `RequestMessage` (author_type=partner). Отклонение или неответ в срок → возврат в `MATCHING` и переход к следующему партнёру `fallback_chain`.
- **FR-5.4** Защита от спуфинга и дублей.

### E6. SLA и автоматизация
- **FR-6.1** SLA принятия и выполнения на **бизнес-часах** (недельный график + IANA-TZ, DST-корректно), паузы на ожидании пользователя/третьей стороны.
- **FR-6.2** Дедлайны/breach считаются на чтении и переходах (работают и без воркера); таймеры — config-gated Dramatiq-актор.
- **FR-6.3** Движок правил `on_create`/`time_based`: автоклассификация→подбор→диспетч, авто-fallback, эскалации.
- **FR-6.4** Метрики: TTFD (до диспетчеризации), TTA (до принятия партнёром), TTR (до выполнения), breach-rate, доля авто-обработанных без человека.

### E7. Приёмка, спор, деньги
- **FR-7.1** Пользователь подтверждает выполнение (`POST /requests/{id}/accept`) → `ACCEPTED_BY_USER`.
- **FR-7.2** Спор (`POST /requests/{id}/dispute`) → `DISPUTE`; при необходимости порождает претензию `COMPENSATION` в kb-support (ссылка `claim_ref`).
- **FR-7.3** Триггер расчёта/escrow/комиссии — вызов платёжного контура; модуль хранит `amount_ref`/`escrow_ref`, **суммы не считает**. Достижение расчёта → `PAID`.
- **FR-7.4** `[ИНВАРИАНТ]` «4 глаза» для любых действий, эквивалентных авторизации выплаты, если такие появляются на стороне модуля.
- **FR-7.5** **Отмена заявки**: пользователь или оператор отменяет заявку из любого нетерминального статуса (`POST /requests/{id}/cancel`, с причиной). Отмена после `DISPATCHED` уведомляет партнёра по каналу и при необходимости отменяет `ServiceOrder`/escrow через контур.

### E8. Уведомления
- **FR-8.1** Пользователю: принято / партнёр назначен / выезд / выполнено / нужна приёмка (push/SMS/email).
- **FR-8.2** Партнёру: новая заявка / изменение / отмена (по каналу + портал LIGHT).
- **FR-8.3** Исходящие webhooks подписчикам (см. §11.4).

### E9. ИИ-агент-оркестратор (agent-ready контур)
- **FR-9.1** Модуль предоставляет агенту **инструменты** — идемпотентные эндпоинты со строгими схемами (создать заявку, классифицировать, подобрать, назначить, диспетчеризовать, перевести статус, запросить контекст). Эндпоинты пригодны как function-tools.
- **FR-9.2** Модуль эмитит **доменные события** (webhooks), на которые агент подписан, чтобы реагировать без поллинга.
- **FR-9.3** **Политика автономности** агента (конфиг): авто-диспетчеризация разрешена только при `confidence ≥ threshold` и доступном канале; иначе — `NEEDS_REVIEW`.
- **FR-9.4** **Human-handoff:** при низкой уверенности, `FAILED_DISPATCH`, споре, претензии — заявка передаётся оператору в kb-support; агент не закрывает такие ветки самостоятельно.
- **FR-9.5** **Guardrails:** агент не инициирует расчёт/выплату сверх контура; «4 глаза» по выплатам сохраняются; все действия агента атрибутируются его сервис-принципалом в `RequestHistory`.
- **FR-9.6** Оркестрация поверх трёх модулей: KB (kb-search) как канал инициации и источник статей; kb-support как контур эскалации/претензий; kb-partners как исполнение партнёрской заявки. Единый паттерн `from-chat`/`from-ticket` → заявка.
- **FR-9.7** **Делегирование прав пользователя (on-behalf-of).** Когда агент действует от имени пользователя, вызовы модуля несут **делегированную авторизацию пользователя** (OAuth2 token-exchange / `actor`-claim); проверки `scope`/`access_level` применяются к **пользователю**, а не к широкому сервис-принципалу агента. Сервис-принципал агента — только для инфраструктурных вызовов.

### E10. Портал партнёра (LIGHT — MVP)
- **FR-10.1** Список заявок партнёра, история; push/email-уведомления.
- **FR-10.2** Приём/отклонение/смена статуса через API/канал (минимальный UI). FULL-дашборд — следующая итерация.

## 9. Каналы доставки — спецификация

### 9.1 Абстракция `DeliveryChannel`
Единый интерфейс (по образцу `LLMProvider` в kb-search), переключение по `PartnerChannelConfig`:

```
interface DeliveryChannel:
    async def deliver(request: ServiceRequest, cfg: PartnerChannelConfig) -> DeliveryResult
    async def parse_inbound(payload, cfg) -> StatusUpdate | None
    def healthcheck(cfg) -> Health
```

Все реализации обёрнуты в resilient-клиент (timeout → circuit-breaker → retry(backoff) → метрики), идемпотентны, секреты берут из kb-vault по ссылке.

### 9.2 Реализации (адаптеры)
| Адаптер | Исходящее (deliver) | Входящее (inbound) | Конфиг |
|---|---|---|---|
| **PartnerApiChannel** | HTTP POST в API партнёра (REST/JSON) | подписанный webhook партнёра | endpoint, auth-ref, схема payload |
| **CrmChannel** | создание заявки в CRM партнёра (адаптеры под популярные CRM РФ — Bitrix24/amoCRM/иные) | webhook/опрос CRM о смене статуса | crm_type, endpoint, auth-ref, маппинг полей |
| **TelegramChannel** | сообщение в чат/бот партнёра (Telegram Bot API) | bot webhook (updates) | bot_token-ref, chat_id, шаблон |
| **MaxChannel** | сообщение в чат партнёра в мессенджере **MAX** (Bot API) `[ДОПУЩЕНИЕ — сверить возможности MAX Bot API]` | bot webhook | bot_token-ref, chat_id, шаблон |
| **EmailChannel** | письмо партнёру (SMTP) с структурированным телом | IMAP-парсер ответов (в РФ) | from/to, шаблон, IMAP-конфиг |
| **MockChannel** | лог/память (для тестов и dev) | синтетические события | — |

> **Расширяемость.** Набор каналов покрывает заявленные варианты (API, CRM, Telegram, MAX, e-mail). Новый канал = новая реализация интерфейса + запись в `PartnerChannelConfig`, без изменения ядра. Реальные SDK/токены сторонних сервисов добавляются через ADR (принцип «разрабатываем сами»).

### 9.3 Выбор и деградация
Для партнёра берётся канал с наивысшим `priority` среди `is_active` и здоровых; при сбое — следующий канал партнёра, затем fallback на следующего партнёра. Все исходы — в `DispatchAttempt` и истории.

## 11. API-контракт (OpenAPI-first)

Сначала пишется `docs/openapi.yaml`, по нему — контрактные тесты. Префикс `/api/v1/partners`.

**Конвенции:** идемпотентные POST принимают заголовок `Idempotency-Key`; формат ошибок — RFC 7807 (`type/title/status`), без ПДн; списки — курсорная пагинация.

### 11.1 Заявки
- `POST /requests` — приём (browser/m2m), идемпотентный.
- `POST /requests/from-chat` — SERVICE-only (kb-search).
- `POST /requests/from-ticket` — SERVICE-only (kb-support).
- `GET /requests` — список, scope-фильтрованный; курсорная пагинация и фильтры (status / category / partner_id / период).
- `GET /requests/{id}` — карточка + `allowed_transitions`, masking по scope.
- `POST /requests/{id}/classify` — (ре)классификация (operator/agent).
- `POST /requests/{id}/assign` — назначение/переназначение партнёра.
- `POST /requests/{id}/dispatch` — запуск/повтор диспетчеризации.
- `POST /requests/{id}/transition` — переход статуса (валидация `allowed`).
- `POST /requests/{id}/messages` — сообщение/внутренняя заметка (`is_internal`).
- `POST /requests/{id}/accept` — приёмка пользователем.
- `POST /requests/{id}/dispute` — открыть спор.
- `POST /requests/{id}/cancel` — отмена заявки пользователем/оператором (с причиной).
- `GET /requests/{id}/requester-context` — operator-only (контекст из rehome.one).

### 11.2 Каналы партнёров (admin/staff)
- `GET/POST /channels`, `GET/PATCH /channels/{id}`, `POST /channels/{id}/healthcheck`.

### 11.3 Входящие от партнёров/мессенджеров
- `POST /inbound/api/{token}` · `POST /inbound/crm/{token}` · `POST /inbound/telegram` · `POST /inbound/max` — подписанные, SERVICE/token-верифицированные, идемпотентные. E-mail — через IMAP-poller (без эндпоинта).

### 11.4 Исходящие webhooks (события)
`request.created / classified / needs_review / assigned / dispatched / dispatch_attempt_failed / dispatch_failed / accepted_by_partner / in_progress / done / accepted_by_user / paid / cancelled / dispute_opened / sla_breached / rated`. HMAC-подпись, доставка после commit.

### 11.5 Служебные
- `GET /healthz` · `GET /readyz` · `GET /metrics` (Prometheus; рассмотреть VictoriaMetrics при росте ретенции).

Все ошибки наружу — `status + title`, без `detail`/ПДн. Недоступный ресурс — `404`.

## 12. Нефункциональные требования и инварианты

`[ОБЯЗАТЕЛЬНО]` для любого модуля reHome:
1. **Отдельный сервис, слабая связанность:** своя БД/репозиторий/деплой; интеграция только по API; никаких shared-таблиц/импортов чужого кода (enforce в CI — `check-arch-constraint.sh`). Сбой модуля не роняет KB/поддержку и наоборот.
2. **«Разрабатываем сами»:** новые внешние зависимости (SDK мессенджеров/CRM/банка) — только через ADR.
3. **Двухконтурность:** `access_level` на каждом ресурсе, фильтр на хранилище, **404, не 403** (anti-enumeration).
4. **scope — только на бэкенде** из проверенного JWT; клиент не передаёт и не вычисляет.
5. **ФЗ-152:** хостинг и обработка ПДн в РФ; маскировка ПДн в логах и **перед любым LLM-вызовом**; soft-delete / right-to-forget; ошибки без утечки ПДн.
6. **Аудит:** значимые действия — неизменяемые записи с `actor_id` (включая сервис-принципал агента).
7. **Внутреннее ≠ внешнее:** внутренние заметки/служебные поля не утекают ни пользователю, ни партнёру.
8. **Идемпотентность** входящих m2m/inbound и POST; **transactional outbox** для исходящих событий и диспетчеризации.
9. **Resilience:** все исходящие клиенты — timeout → circuit-breaker → retry(backoff) → метрики, cache-aside; критичные операции не кешируются.
10. **Объяснимость AI:** решения классификатора и подбора трассируемы (метод/модель/версия/уверенность/причины ранжирования).
11. **Rate limiting / анти-абьюз:** лимиты на публичную форму приёма и на входящие webhooks; защита от replay (timestamp/nonce).
12. **Ретенция ПДн:** `raw_input` и связанные ПДн хранятся ограниченный срок, затем удаляются/обезличиваются (152-ФЗ, right-to-forget); в LLM и логи попадают только `*_masked`-поля.
13. **SLO (ориентир, уточнить):** приём заявки p95 < 300 мс; классификация p95 < 3 с; авто-диспетчеризация инициируется ≤ 5 с после классификации.

## 13. Acceptance-критерии (проверяемые, по эпикам)

Формулируются для контрактных и e2e-тестов; служат гейтами для агента-ревьюера.

- **E1.** Заявка из каждого канала создаётся ровно один раз при повторной доставке (идемпотентность). `from-chat`/`from-ticket` доступны только `kind=SERVICE` (иначе 403/404). `raw_input_masked` не содержит ПДн-паттернов.
- **E2.** При однозначном вводе срабатывает rules-путь без LLM. При `confidence < threshold` заявка уходит в `NEEDS_REVIEW`. `classification` содержит метод/модель/версию/уверенность. Реклассификация фиксируется в истории. Качество классификатора измеряется на golden-наборе kb-eval; точность ≥ целевого порога (уточнить, §16).
- **E3.** `MATCHING` возвращает ранжированный список с `match_trace`; формируется `fallback_chain`. Назначение создаёт/привязывает `ServiceOrder`. Ручное переназначение доступно оператору/агенту.
- **E4.** Диспетчеризация идёт по каналу с верным `priority`/health; каждая попытка — `DispatchAttempt`; повтор доставки не дублирует заказ у партнёра. При неответе/отклонении срабатывает auto-fallback; исчерпание цепочки → `FAILED_DISPATCH` + эскалация. Исходящее партнёру не содержит ПДн сверх состава, разрешённого для категории.
- **E5.** Входящие принимаются только при валидной подписи/токене; дубль не меняет состояние повторно; статусы корректно маппятся на `ServiceRequest`.
- **E6.** Дедлайны/breach считаются без воркера (на чтении/переходах); паузы исключают время ожидания; метрики (TTFD/TTA/TTR/breach/доля авто) доступны.
- **E7.** Приёмка переводит в `ACCEPTED_BY_USER`; спор — в `DISPUTE` и при необходимости создаёт `COMPENSATION` в kb-support; модуль нигде не считает суммы; «4 глаза» соблюдены. Отмена доступна из нетерминальных статусов и при необходимости уведомляет партнёра/отменяет заказ.
- **E8.** Уведомления и webhooks эмитятся после commit; внутренние заметки не уходят наружу.
- **E9.** Все agent-эндпоинты идемпотентны и схематизированы; авто-действия агента подчиняются политике автономности; ветки low-confidence/FAILED/спор/претензия уходят на human-handoff; действия агента атрибутированы в истории. Вызовы агента несут делегированную авторизацию пользователя (on-behalf-of) — доступ не превышает прав пользователя.
- **E10.** Партнёр LIGHT видит свои заявки и может принять/отклонить/сменить статус; чужие заявки невидимы (404).
- **Сквозное.** `check-arch-constraint.sh` зелёный; запрещённый переход → 409; недоступный ресурс → 404; ошибки наружу без ПДн; OpenAPI-валидация в CI зелёная.

## 14. Организация разработки — двухагентная схема

ТЗ структурировано под ту же двухагентную схему, что применялась для KB и kb-support `[сверить точное разделение с вашим процессом]`:

- **Агент-исполнитель** реализует эпик: пишет код строго под OpenAPI-контракт (§11) и доменную модель (§6–§7), сопровождает тестами.
- **Агент-ревьюер/верификатор** («вторые глаза») проверяет реализацию против acceptance-критериев (§13), инвариантов (§12) и контракта; не пропускает эпик, пока гейты не зелёные. Аналог принципа «4 глаза».
- **Контур повторяемости:** OpenAPI-first → контрактные тесты → arch-constraint в CI → программная сборка артефактов с проверкой точных замен (как `build_consolidated.py`).

**Порядок эпиков (предлагаемый):** E1 → E2 → E3 → E4 → E5 → E6 → (E7 ∥ E8) → E9 → E10. E9 (agent-ready) опирается на стабильные контракты E1–E6, поэтому идёт после них.

**Что дать обоим агентам на вход:**
- этот ТЗ + «Бриф `kb-partners`»;
- OpenAPI-контракты kb-platform (`docs/handoff/01_postanovka/04_openapi.yaml`) и kb-support (`docs/openapi.yaml`);
