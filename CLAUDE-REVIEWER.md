## Ревью PR — чек-лист

### Соответствие задаче
- [ ] PR ↔ Issue ↔ FR/NFR; реализация по утверждённому Плану; нет «попутных» правок.

### Качество (анти-костыли)
- [ ] Нет хардкода / закомментированного кода / отключённых тестов / `except: pass`.
- [ ] Нет дублирования (правило трёх), `Any`/`# type: ignore` без обоснования,
      обращений к `_private`, глобального state, магических строк.

### Тесты
- [ ] Unit для нового кода; happy + edge + error; покрытие ≥ 80%; CI зелёный.
- [ ] Integration/контрактные, если применимо (especially маппинг ServiceOrder).

### Безопасность и ФЗ-152
- [ ] ПДн не утекают в логи / LLM / исходящее партнёру (только `*_masked`,
      минимальный состав по категории — FR-4.6).
- [ ] `access_level`-фильтр на уровне хранилища; недоступное → 404, не 403.
- [ ] `scope` — из проверенного JWT на бэкенде, НЕ из payload / фронта.
- [ ] **`is_internal=true` недоступно заявителю и партнёру** — отдельный тест.
- [ ] Действия ИИ-агента атрибутированы; on-behalf-of не превышает прав пользователя.
- [ ] Ретенция `raw_input` соблюдена (NFR-12).

### Идемпотентность и resilience
- [ ] Приём, создание `ServiceOrder`, диспетчеризация — идемпотентны
      (Idempotency-Key / outbox / replay-защита timestamp+nonce).
- [ ] HTTP-клиенты к соседям — timeout → circuit-breaker → retry → метрики,
      graceful degradation (сосед недоступен → kb-partners не падает).
- [ ] SLA-таймеры/outbox — в БД, переживают перезапуск (не в памяти).

### Архитектурная константа
- [ ] CI job `arch-constraint` зелёный. `# arch-allow:` — reason ≥10 chars и легитимен.
- [ ] Нет прямых SQL к чужим БД / импортов кода соседей (в т.ч. через dynamic import).
- [ ] Клиенты к external API — в `backend/src/api/clients/`.

### Git и процесс
- [ ] Ветка/коммиты по правилам (Conventional Commits, scope из списка).
- [ ] PR ≤ 800 строк или обоснование; шаблон заполнен; CI зелёный.

## Решение по PR

- **approve** — все пункты закрыты, можно merge.
- **request-changes** — каждое замечание отдельным комментарием со строкой и
  пунктом правила.
- **reject** — задача поставлена некорректно / PR нереализуем; эскалация Архитектору.

## Критичные модули (повышенная тщательность, однофазный коммит не применяется)

- **`api/requests/`** — ядро + FSM. Баг с `access_level`/`is_internal` — утечка.
- **`api/classifier/`** — маскировка ПДн перед LLM; трассируемость решений.
- **`api/channels/` + `api/dispatch/` + `api/inbound/`** — идемпотентность,
  подпись/replay-защита входящих, отсутствие ПДн сверх состава категории.
- **`api/agent/` (E9)** — политика автономности, human-handoff, on-behalf-of.
- **`api/clients/`** — circuit breaker / graceful degradation; маппинг ServiceOrder.

## Помни главное

Двухагентная схема работает на дисциплине. Один пропущенный костыль = прецедент =
техдолг. Архитектурная константа (отдельный сервис) — инвариант, не рекомендация.
Будь требовательным и конкретным; эскалируй спорное.
